### 1. 批量请求 + 并发控制

大部分初学者会一个接一个地发请求（串行），导致连接闲置期长、Token 连续计费。正确做法是批量组装上下文，用 `Promise.all` 配合限流器并发请求。

javascript
// 串行示例（不推荐）
async function askGrok4(prompt) {
  const response = await fetch('https://www.qianjuai.com/v1/chat/completions', {
    method: 'POST',
    headers: { 'Authorization': 'Bearer YOUR_KEY' },
    body: JSON.stringify({ model: 'grok-4', messages: [{ role: 'user', content: prompt }] })
  });
  return response.json();
}

// 批量并发（推荐）
async function batchAskGrok4(prompts, concurrency = 5) {
  const results = [];
  for (let i = 0; i < prompts.length; i += concurrency) {
    const batch = prompts.slice(i, i + concurrency);
    const batchResults = await Promise.all(batch.map(p => askGrok4(p)));
    results.push(...batchResults);
  }
  return results;
}

### 2. 使用流式响应减少首字节延迟

Grok4 的流式输出能力很强，但很多企业却用非流式模式等整个回复返回，白白浪费等待时间。改成 `stream: true`，搭配 `EventSource` 或 `async iterator`，用户能实时看到部分结果，且服务端可以提前释放连接池。

javascript
const response = await fetch('https://www.qianjuai.com/v1/chat/completions', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json', 'Authorization': 'Bearer YOUR_KEY' },
  body: JSON.stringify({ model: 'grok-4', messages: [...], stream: true })
});
const reader = response.body.getReader();
const decoder = new TextDecoder();
let partial = '';
while (true) {
  const { done, value } = await reader.read();
  if (done) break;
  partial += decoder.decode(value, { stream: true });
  // 处理 SSE 事件
}

### 4. 上下文窗口压缩

Grok4 上下文窗口很长（128K），但多数对话根本不需保留全部历史。每次请求前只保留最近的 3 轮对话，加上系统 prompt 里明确限制 token 长度。还可以用滑动窗口算法动态裁剪。

javascript
function compressMessages(history, maxRounds = 3) {
  return history.slice(-maxRounds * 2).filter((m, i) => i % 2 === 0 || i === history.length - 1);
