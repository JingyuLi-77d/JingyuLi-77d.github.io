## 第一大坑：用了“非标准”的Java库，你会被重试机制和异步逻辑“背刺”

很多 Java 开发者，习惯了 Spring Boot 生态，习惯性地去找一个封装好的 `mistral-sdk-java` 或类似名字的依赖。一搜，GitHub 上星不少，文档也全，直接 `maven` 引入。你以为是“开箱即用”，其实是“开箱即坑”。

**坑点 1：非官方的库，重试策略是你不可知的“黑洞”。**

某些第三方库，为了“保证”请求成功率，默认开启了一个**指数退避重试**机制。Mistral 的 API 如果在第一次调用时遇到 500 或 429 状态码，这个库会自动在背后重试 3 次甚至 5 次。你以为是跑一次请求，结果它替你跑了 4 次。账单怎么可能不翻倍？关键是，你自己代码里没写任何重试逻辑，完全不知道什么时候触发了这个“隐藏”行为。

**坑点 2：异步处理和回调，实际用起来是“噩梦”。**

非官方的库，尤其是那些为了“高性能”自己撸了异步 I/O 的，往往会滥用 `CompletableFuture` 或者自建的线程池。你写一个看起来是“同步”的调用，实际它在底层维护了一个线程池。当你跑高并发时，这个线程池的容量、超时配置、队列溢出策略，全是默认参数。轻则请求超时被丢弃，重则线程池溢出，导致 JVM 直接崩溃。

## 第二大坑：死连接+长连接池，你的 Java 服务在 30 分钟后开始“吃错账”

Java 接入 API 另一个普遍问题，是**连接池的管理失误**。很多人图方便，直接用 `HttpClient` 的默认连接池，或者用了一个连接池设置不当的第三方库。

**坑点 3：连接的“假活”状态。**

Mistral API 为了安全，一般在空闲 30 分钟-1 小时后，会主动关闭 TCP 连接。但你的 Java 应用（或者你用的那个库）并不知道。它以为连接还是好的，于是拿着这个“死连接”去发请求。结果就是，请求被莫名其妙地阻塞、超时，然后你的重试逻辑又启动，再创建一个新连接。一来一回，一个请求可能花了 5 秒才完成，吞吐量直线下降，加上重复请求，账单“唰”地就上去了。

**坑点 4：连接池被撑爆，导致服务注册中心心跳中断。**

更严重的是，在高并发的场景下，大量的“死连接”占着连接池的坑位。新的请求来了，连接池没有可用连接，只能创建新连接。到最后，连接数超过了系统限制（`ulimit` 或者系统 `net.ipv4.tcp_tw_reuse` 配置），Java 服务直接报 `Too many open files`。这会波及到注册中心（比如 Eureka 或 Nacos）的健康检查，导致服务被误判为宕机，整个调用链路出问题。

**正确的姿势是什么？**

务必在 Java 代码中显式配置连接池的**健康检查**和**空闲连接回收**。如果你用 `Apache HttpClient` 4/5，标准配置如下：

java

// 关键：设置空闲连接验证时间
connManager.setValidateAfterInactivity(5000); // 5秒后验证连接是否“活”

// 关键：设置空闲连接回收线程
IdleConnectionEvictor idleConnectionEvictor = new IdleConnectionEvictor(connManager, 5, TimeUnit.SECONDS);
idleConnectionEvictor.start();
